# Git分支管理与提交规范

## 一、分支模型

采用 Git Flow 分支管理策略，分支类型如下：

| 分支 | 用途 | 命名示例 |
|------|------|----------|
| master | 生产环境代码，仅通过merge更新 | master |
| develop | 开发主分支，日常开发代码合并到此 | develop |
| feature | 新功能开发 | feature/user-login |
| bugfix | 修复develop分支上的Bug | bugfix/order-price-error |
| hotfix | 紧急修复生产环境问题 | hotfix/payment-crash |
| release | 发布准备分支 | release/v2.1.0 |

## 二、分支操作流程

1. 从develop创建feature分支：`git checkout -b feature/xxx develop`
2. feature分支开发完成后合并到develop：提交PR → Code Review → merge
3. 准备发布时从develop创建release分支：`git checkout -b release/vX.X.X develop`
4. release分支测试通过后合并到master和develop
5. 紧急修复从master创建hotfix分支，修复后合并到master和develop

## 三、Commit提交规范

### 格式

```
[类型] 简要描述（不超过50个中文字符）

[可选的详细描述]
```

### 类型标识

| 类型 | 说明 |
|------|------|
| feat | 新功能 |
| fix | Bug修复 |
| refactor | 代码重构 |
| style | 代码格式调整 |
| docs | 文档更新 |
| test | 测试相关 |
| chore | 构建/工具配置变更 |
| perf | 性能优化 |

### 示例

```
[feat] 用户登录增加短信验证码功能

- 集成阿里云短信服务SDK
- 支持60秒发送间隔限制
- 每日单用户上限10条
```

## 四、Code Review规范

1. 所有合并到develop和master的代码必须经过Code Review
2. 至少1位同组开发人员Approved后方可合并
3. Reviewer须检查：代码逻辑正确性、是否符合编码规范、是否存在安全风险、测试覆盖是否充分
4. PR应控制在500行以内，超过则拆分为多个小PR

## 五、Tag管理

1. 生产环境每次发布必须在master分支上打Tag
2. Tag命名格式：v<主版本>.<次版本>.<修订号>，例如 v2.1.0
3. 主版本号：重大功能变更或不兼容的API修改
4. 次版本号：向下兼容的功能新增
5. 修订号：向下兼容的Bug修复
